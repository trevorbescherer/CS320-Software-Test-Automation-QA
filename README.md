# CS320-Software-Test-Automation-QA
1. How can I ensure that my code, program, or software is functional and secure?
The most reliable way I've found to ensure functionality is to write tests that are tied directly to requirements,not just tests that confirm the code does what I think it should do, but tests that verify it meets the actual specified behavior. In the Contact Service project, this meant writing unit tests that validated every constraint defined in the requirements: field length limits, null rejection, ID immutability, and correct behavior on add, update, and delete operations. Keeping test coverage tight to requirements means that if a requirement is met, there's a test proving it, and if something breaks, a test catches it. For security specifically, defensive programming practices like input validation and rejecting null or malformed data at the boundary of each class prevent a wide class of issues before they can propagate deeper into the system.

2. How do I interpret user needs and incorporate them into a program?
User needs rarely arrive as clean technical specifications. The work is translating intent into concrete, testable requirements. In this course, the project requirements described what the contact service must do — enforce field constraints, prevent duplicate IDs, allow specific fields to be updated, and my job was to make sure those constraints were fully understood before writing a single line of implementation code. Writing the tests first, or at least defining them alongside the implementation, helped me stay honest about whether each requirement was actually being met rather than assumed. I've found that when requirements are ambiguous, it's better to make the assumption explicit in a test than to leave it implicit in the code, because at least then the assumption is visible and reviewable.

3. How do I approach designing software?
I approach design by starting with the boundaries of each component (what it accepts, what it rejects, and what it promises to return) before thinking about internal implementation. For the Contact Service, that meant defining the Contact class's constraints first, then designing ContactService around the operations it needed to support, and only then filling in the logic. This outside-in approach keeps the design honest because the interface has to make sense before you commit to an implementation. Testing reinforces this: if a class is hard to test, it's usually a signal that the design has too many responsibilities or too many hidden dependencies. Writing testable code and writing well-designed code turn out to be largely the same discipline.
